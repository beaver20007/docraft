# engine-001e — eval-харнесс (recall + false-positive) + конфигурируемый профиль модели

**Submodule:** `apps/ai-playbook` → `beaver20007/ai-playbook-generator`, каталог `apps/api`
**Класс:** YELLOW. Мерж — только Оркестратор.
**Статус:** QUEUED → В РАБОТЕ. **Последняя под-задача спринта.** Строит на всём влитом:
`001a`-`001d`, `quality-001/002/003`.

## Цель

Измеримо оценивать гейт качества (механический валидатор R1-R6 + LLM-судья + repair) на
эталонном корпусе, и заложить конфигурируемый профиль модели по стадиям. Это база для решения
«включать ли blocking» и для регресса при будущих правках промптов.

## Рамки (заданы владельцем 2026-07-21 — три правки к ТЗ спринта, исполнять как есть)

### 1. ДВЕ метрики: recall И false-positive rate (главное дополнение)
- **Дефектный корпус** (кейсы с известными дефектами) → **recall** = доля пойманных известных
  дефектов.
- **ЧИСТЫЙ КОРПУС (обязательно, сейчас его нет):** документы с КОРРЕКТНЫМИ, согласованными
  числами, где ожидание = **НОЛЬ находок**. → **false-positive rate** = находки на чистом корпусе.
- Eval выдаёт ОБЕ цифры. **Без FP-цифры blocking включать не на чем** — это критерий готовности.
- Чистый корпус — не одна фикстура: разные формы (таблицы с «Итого», воронки со СХОДЯЩИМИСЯ
  процентами, KPI без противоречий, кириллица, много позиций) — чтобы FP мерился на реальном
  разнообразии, а не на одном тепличном примере.

### 2. РЕАЛЬНЫЙ прод-документ как регресс-фикстура (job caee3bb7)
Ценнее синтетики — проверка, чинит ли пайплайн то, что сам выдал в прод. Настоящие дефекты:
- «25% конверсия лид→сделка» против произведения этапов 30-40%×60-70%×50-60%×70-80% ≈ 9-13%
  (кросс-метричная арифметика → **судья**);
- «80 лидов/день» × «30% конверсия» ≈ 480 встреч/мес против плана «15 встреч/мес» (**судья**);
- churn «<10%» в метриках против «<8%» в KPI CSM (**механика R5**, quality-003);
- «MODULE 01» латиницей (**E2**, локализация — вне числового гейта; зафиксировать как известный
  промах числового слоя, ловится отдельной E2-проверкой).
Собрать IR, воспроизводящий эти дефекты (реальный прод-IR недоступен — реконструкция по описанию;
если владелец достанет — заменить). Ожидание по каждому дефекту явно: чем должен ловиться.

### 3. Профиль модели — конфигурируемый, но ВСЕ стадии на SONNET (изменение указания из ТЗ спринта)
- **НЕ удешевлять План/Черновик.** Построй механизм per-stage (конфигурируемый — через
  settings/env, не хардкод), но ВСЕ стадии оставь на **Sonnet** по умолчанию.
- Причина та же, что с судьёй: удешевление бьёт по качеству, ради которого спринт затеян; трафик —
  один тестовый аккаунт, экономия копеечная. Переключать стадию на дешёвую модель — только когда
  eval покажет, что качество не падает.
- Т.е. дефолт профиля генерации = Sonnet на всех стадиях; механизм позволяет переопределить
  стадию конфигом, но по умолчанию ничего не удешевлено. `numeric_judge` = Sonnet (001d) не менять.

## Про LLM и детерминизм
- Механический валидатор (R1-R6) — детерминирован, recall/FP по нему меряются БЕЗ LLM, точно.
- Судья (Sonnet) — реальный recall/FP требует боевого ключа. Харнесс должен: (а) уметь прогон
  судьи на корпусе при наличии `ANTHROPIC_API_KEY` (без ключа — пропуск/skip, честно пометить);
  (б) в юнит-тестах мокать судью (детерминизм, как в 001d). Не выдавать мок-цифры за боевые.

## DoD
- Eval-харнесс гоняет корпус, выдаёт **recall (дефектный) + FP-rate (чистый)** двумя числами.
- Чистый корпус разнообразен; дефектный включает калибровочные кейсы (3.2 касаний vs 8-12,
  MRR/ARR, 85/90, MODULE латиницей) + **реальный кейс caee3bb7** с явными ожиданиями.
- Профиль модели конфигурируем per-stage, дефолт = **Sonnet везде** (ничего не удешевлено).
- Тесты `apps/api` зелёные ДО и ПОСЛЕ (прогон с выводом; LLM мокать).
- Отчёт приводит фактические recall/FP по механическому слою (детерминированно) и описывает,
  как гонять судью на ключе.
