# engine-001 — движок: монолит → стадии + продающий playbook
### Спринт (не один PR). Submodule: `ai-playbook`. Разбить на под-задачи ниже.

Цель разморожена: глобальное улучшение движка на базе каталога скиллов, основа —
текущий playbook + разбор его качества + рубрика. Тип-стратегия: довести ОДИН реальный
тип (playbook) до «продаётся» + слой, где новый тип = конфиг, не копия промпта.

---

## Не-цели (жёсткие границы)
- **LangGraph не вводить** — стадии чистыми функциями/модулями, чтобы переход на граф позже был дёшев.
- Монетизация / Stripe / тарифы — не здесь.
- Веб / UI — не здесь (это `web-001` / Трек 2).
- **OCR не трогать** (magical-pdf).
- Координация с влитым: `quality-002` уже подключил валидатор (observe-and-log) и починил fail-open в `quality_service.py` — repair-loop строится **НА нём**, не переписывает.

## Целевой разрыв (из экспертного разбора playbook'а)
Sonnet-база = «B+ дженерик». До «продаётся» закрыть четыре бреши: **специфичность,
консистентность, плотность ассетов, локализация.** Движок бьёт по каждой конкретной стадией + гейтом.

---

## Архитектура — стадии (из монолитной Celery-задачи)
Явные стадии, IR-контракт на выходе уже есть. Единая анатомия промпта на каждой
(System-structure): роль / контекст / вход / формат вывода / границы.

1. **План** — структура документа под тип + вход. Chain-of-thought: рассуждение скрытое, наружу — только структура.
2. **Сбор/обогащение** — втянуть специфику клиента (продукт, ICP, чек, рынок, размер команды) в рабочий контекст. Это фундамент специфичности.
3. **Черновик секций** — генерация по шаблону типа; специфику **вплетать**, а не писать «адаптируйте под свой продукт».
4. **Форматирование → IR** — Constraint-spec: строгая валидация выхода по IR-схеме (ноль свободного текста туда, где ждём структуру).
5. **Гейт качества** — расширяет `quality-002`: механический `quality-001` (влит) + LLM-судья + **repair-loop**.

## Template-design — типы как конфиг
- Вынести playbook из текста промпта в **параметризованный шаблон**: именованные слоты,
  правила заполнения, per-секция требования (какие ассеты обязательны, минимальная плотность).
- Playbook довести до отличного; **новый тип = новый конфиг шаблона**, не копия промпта.
- Фейковые enum-«типы» (сейчас 1 playbook под видом 3) — скрыть/убрать до появления реальных.

## Закрытие брешей качества (маппинг на рубрику docs/quality-rubric)
- **Специфичность (C):** стадия Сбора обязана втянуть вход-параметры; стадия Черновика — вплести. Гейт C1/C2 (LLM-судья) валит дженерик и плейсхолдер-инструкции в теле.
- **Консистентность (A):** механический `quality-001` уже ловит противоречия цифр/единиц (влит); движок добавляет A2 (смысловая консистентность, LLM-судья) в repair-loop.
- **Обоснованность (B):** бенчмарки — из входа, либо с сегментом-квалификатором, либо убрать.
- **Плотность ассетов (D):** per-секция минимум ассетов задаётся в шаблоне; упомянутый ассет обязан быть дан (D1). Черновик разворачивает, гейт проверяет.
- **Локализация (E):** политика языка/глоссария — стилевой констрейнт в System-structure; E2 (латиница в лейблах) уже механически влита.

## Тон и голос
- **Tone-calibration:** тон входным параметром документа (формальный↔тёплый), дефолт по типу.
- **Persona-architecture:** голос от лица компании с ограничениями характера, чтобы не звучало роботом.

## Repair-loop (в гейте)
- `error` → авто-ремонт проблемной секции (пере-промпт стадии Черновика/Форматирования по **конкретной** находке) → пере-проверка → ≤N итераций → при неуспехе **мягкое состояние** (Frustration-checks), не показ брака.
- Флаг «blocking» включить, когда repair-loop готов и обкатан. **До этого — режим observe-and-log** (как сейчас), генерацию не рвать.

## Профиль модели по стадии
- План/Черновик — можно дешевле; Форматирование/финал — Sonnet. Прод уже на premium; здесь тонкая настройка per-стадия под стоимость/качество.

## Eval-харнесс
- Эталонные входы, включая краевые кейсы (Few-shot): пустые поля, длинные названия, кириллица, много позиций в смете.
- → снапшоты IR-вывода + прогон гейта.
- **Калибровочные кейсы из разбора playbook'а как регресс-тесты:** «3.2 касаний» vs «8-12», смешанные MRR/ARR, точность 85%/90%, «MODULE» латиницей.
- Правки промптов не должны ронять eval.

## Под-задачи спринта (каждая — свой PR, аудит базы, локальный прогон; CI красный осознанно)
1. **`engine-001a`** — извлечь стадии из монолита (структурно, **поведение сохранить**, тесты зелёные). Никаких новых промптов — чистый рефакторинг в стадии.
2. **`engine-001b`** — template-layer: playbook в конфиг; фейк-типы скрыть; per-секция требования к ассетам.
3. **`engine-001c`** — System-structure по стадиям + Tone-calibration + Persona-architecture.
4. **`engine-001d`** — LLM-судья (A2/B/C/D) + repair-loop поверх `quality-002` (дешёвая проверочная модель на извлечённых фактах, не на всём документе).
5. **`engine-001e`** — eval-харнесс + профиль модели по стадии.

Порядок жёсткий: 001a (рефактор) до всего; 001d строится на 001a-стадиях и влитом `quality-002`.

## DoD (спринта)
- Генерация идёт явными стадиями; поведение при 001a не изменилось (тесты).
- Playbook в конфиг; новый тип добавляется конфигом; фейк-типы убраны.
- Гейт с repair-loop закрывает A/B/C/D/E; тон переключается параметром.
- Eval зелёный, включая калибровочные кейсы.
- На реальном входе документ проходит рубрику без блокеров и **субъективно тянет на «продать»** (глазная проверка владельца против сегодняшнего B+ бенчмарка).
- OCR и рендер не задеты; каждая под-задача — локальный прогон с выводом (CI облачный красный по политике).

Первой командой в каждой под-задаче: `pwd` + `git branch --show-current`.
