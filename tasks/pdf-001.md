## Проблема (подтверждена чтением кода, не отчётом)

`packages/render-engine/render_engine/fonts.py` регистрирует ровно два TTF-имени: `Arial`,
`Arial-Bold` (+ встроенный `Courier`). `resolve_font()` знает фоллбэки только для них.

Темы просят другое:
- `themes/academic-serif/theme.json:36-38` → `Georgia-Bold`, `Georgia`, `Courier`
- `themes/premium/theme.json` (`typography`) → `Inter-Bold`, `Inter`, `Monaco`

`resolve_font("Inter")` не находит имя ни в `_registered`, ни в `fallbacks` → возвращает `"Inter"`
как есть → ReportLab получает незарегистрированное имя шрифта. Итог: падение рендера либо
молчаливо неверный шрифт. Обе темы — `premium` в том числе (платный тир) — под ударом.

Дополнительно: базовый Helvetica/Courier из PostScript **не имеет кириллицы**. Если фоллбэк
уводит в них, русский текст в PDF выйдет пустыми глифами.

## Цель (DoD)

1. Любое логическое имя шрифта из любой темы в `themes/*/theme.json` разрешается в реально
   зарегистрированный, **кириллический** шрифт — без исключений и без «квадратиков».
2. `fonts.py` расширен: serif-семейство (Georgia → DejaVu Serif / Liberation Serif),
   sans (Inter → DejaVu Sans / Liberation Sans), mono (Monaco → DejaVu Sans Mono /
   Liberation Mono) — с кандидат-путями для Windows **и** Linux (прод — Docker/Railway).
3. `resolve_font()` больше не может вернуть незарегистрированное имя. Неизвестное имя →
   детерминированный кириллический фоллбэк + `warn` в лог, не молчание.
4. **Проверка прогоном, не рассуждением:** сгенерировать PDF по каждой из 7 тем
   (`themes/*/theme.json`) с текстом, содержащим кириллицу, латиницу и цифры; извлечь текст
   обратно (`pdfminer`/`PyPDF2`) и убедиться, что кириллица читается. Вывод команд —
   в отчёт. **Особое внимание `academic-serif` и `premium`.**
5. Регрессионный тест в `packages/render-engine` (или `apps/api/tests/`), который падает
   на старом коде и проходит на новом: `resolve_font` для всех имён из всех тем возвращает
   зарегистрированное имя.
