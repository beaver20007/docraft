# web-001 — UI не отличает «проверка провалена» от «проверка не выполнилась»

**Submodule:** `apps/ai-playbook` → репозиторий `beaver20007/ai-playbook-generator`, каталог `apps/web`
**Класс:** YELLOW (правка фронта + PR). Мерж — только Оркестратор.
**Статус:** QUEUED. Не блокирует ничего, но чем дольше висит — тем дольше в UI дыра.
**Ветка:** `fix/qa-errored-checks-ui` от свежего `origin/main`.

## Откуда взялась задача

Трек `quality-002` (влит `34dff6a`) починил fail-open в `apps/api/app/services/quality_service.py`:
раньше упавшая LLM-проверка возвращала `passed=True, score=8` — неотличимо от честно пройденной,
и UI рисовал зелёную галочку там, где не выполнилась ни одна проверка. Теперь у `CheckResult`
есть `executed: bool` и `error: str | None`, а в сводке джобы `checks_errored` отделён от
`checks_failed`; в `check_details` добавлены `executed` и `error`.

**Фронт про это не знает.** `apps/web/components/PreviewGallery.tsx` строит бейджи из
`checks_passed` / `checks_failed`. Проверка, которая не выполнилась, теперь не попадает
ни в один из этих списков → отрисуется как «нет данных». Это лучше прежней зелёной галочки,
но всё ещё неверно: пользователь не видит, что контроль качества не отработал.

## Цель (DoD)

1. UI различает три состояния проверки, а не два: **пройдена** / **провалена** /
   **не выполнилась** (с причиной из `error`). Третье состояние визуально ближе к тревоге,
   чем к успеху — но не выдаёт себя за провал документа: документ не проверен, а не плох.
2. Общий вердикт по документу не показывается зелёным, если хоть одна проверка не выполнилась.
3. Проверить **фактическим прогоном в браузере**, а не по коду: поднять веб, получить
   состояние джобы с непустым `checks_errored` (проще всего — замокать ответ API), показать
   скриншот всех трёх состояний. Инструмент превью для этого есть.
4. Ничего не ломать в существующем отображении пройденных/проваленных проверок —
   регрессию проверить тем же прогоном.
5. Заодно посмотри, нет ли рядом такого же «отсутствие данных трактуется как успех» —
   в `ir_numeric` (новое поле сводки от `quality-002`) статус `validator_error` с
   `passed=null` не должен где-либо читаться как «чисто». Найденное — в отчёт;
   чинить, только если это тот же файл.
