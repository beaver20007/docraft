## 7. Подводные камни (10, из HANDOFF-manual.md — читать перед действием)

1. **Прямой push в `main` заблокирован** auto-mode — вся работа через ветку + PR (включая правки этого файла).

8. **Node/npm ломались** (`nvm use` без npm) — лечится `nvm uninstall/install 24.18.0`. Симптом: `spawn npx ENOENT`.
9. **Консоль Windows в cp1251** калечит кириллицу — проверять текст записью в UTF-8 файл, не `print` в консоль.

## 10. Политика проверки качества (решение владельца 2026-07-21 — не пересматривать без него)

**Это не сбой инфраструктуры, а сознательный выбор.** Следующая сессия не должна поднимать
этот вопрос заново и не должна эскалировать красный CI как инцидент.

1. **Облачный GitHub Actions на платном биллинге не используем.** Биллинг чинить **не надо**.
2. **Красный статус облачных прогонов Actions — ОЖИДАЕМ** и **не является сигналом о состоянии
   кода**. Не путать с настоящей поломкой тестов. Повторно не эскалировать.
3. **Self-hosted раннер для docraft не поднимаем.** Managed-деплой (Railway/Vercel) плюс лёгкий
   pytest — раннер здесь оверкилл. Модель Stratum сюда **не переносится**.
4. **Единственный валидный сигнал качества — локальный прогон тестов с выводом команд.**
   Это относится и к Оркестратору, и к воркерам: в каждом ТЗ должно быть сказано, что
   красный чек ничего не значит, а доказательство — приложенный вывод.
5. **Перед каждым необратимым шагом** (мерж / деплой / удаление) — проверка базы и содержимого
   **фактом**, даже при санкции владельца. Санкция снимает вопрос «делать ли», но не отменяет
   проверку «то ли делаем».
6. **«PR смержен» ≠ «работает на реальности».** Отслеживать живой результат: на проде, в образе,
   в браузере — а не git-статус. Пример этой сессии: `pdf-001` смержен, но `Inter` в образе так
   и не появился до `fonts-001` — тема `premium` продолжает рендериться Arial'ом.
7. **Факт против основания решения → стоп, к владельцу.** Если проверка показывает не то,
   на чём решение строилось, — не «дожимать по плану», а остановиться и доложить.

### Прецеденты, где это уже сработало в этой сессии

- Красный CI на PR #7 выглядел как упавшие тесты — annotation показал биллинг Actions.
  Мержить на основании «зелёного бейджа» было бы нельзя, но и краснота была не про код.
- Мой собственный сравнительный прогон `pdf-002` на `main` упал с `ValueError: Can't map
  determine family/bold/italic for arial` — это оказался артефакт моего стенда (в изоляции
  никто не вызвал `register_fonts()`), а не прод-баг. Прогон переделан честно, и только тогда
  расхождение `#888888` против цвета темы стало фактом.

## 11. Инцидент с токеном (2026-07-21, сессия-2) — ЗАКРЫТ

**Что было.** В `apps/ai-playbook/.git/config` remote-URL содержал GitHub PAT открытым текстом:
`https://beaver20007:ghp_…@github.com/beaver20007/ai-playbook-generator.git`. Токен печатался
в выводе любой git-команды и оседал в логах агентских сессий. Заражён был **один remote из
шести** — суперрепо, `magical-pdf`, `desktop-instructor` чистые.

**Как опознали токен.** GitHub у classic PAT не показывает даже часть значения, поэтому
сопоставление шло по поведению, а не по виду:
1. `X-OAuth-Scopes: repo, workflow` (запрос к `api.github.com/user` самим токеном) — отсекло
   4 кандидата из 8, у которых только `repo`.
2. Само это обращение пометило нужный токен свежим «last used» — владелец обновил страницу
   и увидел, у кого из четырёх отметка изменилась. Это оказался **`docraft-push`**.

**Что сделано.** Владелец ротировал `docraft-push`. Перед перезаписью проверено, что старое
значение мертво: `HTTP 401`. Remote переписан на чистый URL; аутентификация идёт через уже
настроенный глобальный credential helper `gh auth git-credential` — встроенный токен был
**избыточен с самого начала**.

**Проверки после.** `git fetch` проходит (exit 0) из submodule и из обоих worktree воркеров;
все 6 remote чистые; `ghp_` в git-конфигах не встречается.

**Побочная находка.** Значение токена всплыло в `apps/web/.next/dev/cache/turbopack/*.sst`
(кеш Next.js dev-сервера воркера `web-001`, 153 МБ). В git не попал — `.next` закрыт
`apps/web/.gitignore:17`, в коммитах ветки и в диффе PR совпадений ноль. Кеш удалён.

**Уроки.**
1. Секрет в remote-URL не остаётся локальным: он попадает в вывод команд, в транскрипты
   агентов и в кеши сборки — то есть расползается по путям, за которыми никто не следит.
2. `.gitignore` спас от утечки в репозиторий, но не от появления секрета на диске в третьем
   месте. При ротации мало переписать конфиг — надо искать значение сплошным поиском.
3. Credential helper через `gh` уже был настроен глобально. Токен в URL не давал ничего,
   кроме риска. **Новых remote с токеном в URL не заводить.**

## 13. E2E_SKIP_AUTH — обход авторизации на проде (2026-07-21, сессия-2) — ЭСКАЛАЦИЯ, H16

Найдено при осмотре `apps/web` по заданию владельца (флаг выставлен в Vercel Production).
**Оркестратор находку НЕ чинил** — это RED (авторизация + прод): разобрался, задокументировал,
эскалировал. Правило «инцидент: сначала стоп, потом диагностика» здесь неприменимо — стоп
(снятие переменной с прода) в руках владельца, у Оркестратора прод-доступа нет.

**Что делает флаг (проверено кодом и доками Next):**
- `apps/web/proxy.ts:10-12`: `AUTH_ENABLED = Boolean(SUPABASE_URL && ANON_KEY) && E2E_SKIP_AUTH !== "true"`.
- `proxy.ts:28-30`: при `!AUTH_ENABLED` — `return baseResponse` для **любого** пути, без
  проверки сессии («allow everything through»).
- Значит `E2E_SKIP_AUTH="true"` в проде = **весь сайт доступен без входа**.

**Почему `proxy.ts` — это активная защита, а не мёртвый код:**
- В репо нет `middleware.ts`; `proxy()` нигде не импортируется — на первый взгляд похоже на
  отключённый код. Но история (`git log --follow`): `middleware.ts` (Supabase-auth, `aac583b`)
  → в него добавлен `E2E_SKIP_AUTH` (`be4f47b`) → **`6818fe7` переименовал `middleware.ts` →
  `proxy.ts`** «под Next.js 16 proxy convention».
- Проверено по офиц. источникам (блог Next.js 16, Auth0, GitHub-дискуссия vercel/next.js):
  **Next.js 16 действительно сделал `proxy.ts` + `export proxy()` штатной заменой middleware.**
  То есть переименование корректно, `proxy.ts` — активный перехватчик на границе сети.

**Радиус (оценён, не полон):**
- Клиентские `getUser()` в `app/page.tsx` / `app/history/page.tsx` — UX-гейты, обходятся,
  не безопасность.
- `lib/generation/authHeaders.ts` шлёт токен сессии в backend API — **если** FastAPI его
  проверяет, данные за API защищены независимо от proxy. **Не проверял — не утверждаю.**
- SERVICE_ROLE-ключа в вебе нет → чтения Supabase идут под anon-ключом, RLS применяется
  независимо от proxy. То есть БД, вероятно, не нараспашку; но маршрутный доступ к приложению
  при `true` — открыт.

**Вопрос владельцу (H16):** какое значение у `E2E_SKIP_AUTH` в Vercel Production и зачем
тестовый флаг живёт в прод-конфиге? Если `"true"` — снять (правило 5 подводных камней: удаление/
пересоздание прод-переменных Vercel — только с явного разрешения владельца, простое отсутствие
даёт `branch_not_found`, поэтому менять должен владелец). Флаг введён давно, не относится к

## 15. Находка по валидатору `[ir-numeric]` (2026-07-21, сессия-2) — разобрано фактом

**Повод.** Прод job `caee3bb7-641e-48a5-92f4-896ec7900bb4`: `[ir-numeric] passed=True
errors=0 warnings=0 by_rule={} rules=-`, а документ содержал реальные противоречия
(воронка 9-13% vs 25%; 80 лидов×30% vs 15 встреч/мес; churn <10% vs <8%; «MODULE 01» латиницей).

**Разбор Оркестратора (чтение кода + прогон настоящего `ir_numeric_validator` на реконструкции):**

1. **`by_rule={}` / `rules=-` ≠ «правила не зарегистрированы».** R1-R6 зарегистрированы в
   `_RULES` и прогоняются в `validate_numeric_consistency`. `by_rule` строится ИЗ найденных
   findings (`quality_gate.py:116`); `rules=` = `join(...) or "-"`. Пусто = **ноль findings**,
   не ноль правил. Правила слепыми НЕ являются — но их охват узкий и механический.
2. **Что валидатор ловит (подтверждено прогоном):** R5 поймал `churn 10 vs 8` (warning),
   когда метки в двух `stats_grid` идентичны.
3. **Что НЕ ловит и почему:**
   - воронка-произведение, `80 лидов×30%` — **семантические кросс-метричные выводы**, R1-R6
     такой арифметики не делают by design («промах дешевле ложного срабатывания»). Это работа
     LLM-судьи `engine-001d`.
   - «MODULE 01» — правило **E2 (локализация)**, в числовом валидаторе его нет вовсе.
   - churn в реальном доке — **R5 промолчал**: метка/структура не совпали (метрика в прозе/
     KPI-карточке, а не в двух stats_grid). Настоящая **калибровочная брешь R5**.

**Вывод — не баг, а граница + калибровка.** `[ir-numeric] passed=True` значит только «нет
механического нарушения R1-R6», НЕ «числа верны». **Опасность:** если `engine-001d` построит
гейт, трактуя mechanical-pass как semantic-pass, — противоречия уедут в прод.

**Действие (по указанию владельца «заведи задачу на калибровку ДО гейта»):** заведён трек

## 17. Разбор роста кегля в таблицах/карточках (2026-07-21, сессия-2) — факт

Владелец уточнил: «Sales Manager» тот же столбец — старый PDF (DejaVu) 58.9pt одной строкой,
новый (Inter) 44.9+39.4=84.3pt разорвано. +43% ширины при гарнитуре на 6% УЖЕ → **вырос кегль**,
не ширина шрифта. Гипотеза владельца: причина — `pdf-002` (реальные токены вместо захардкоженных).

**Проверка Оркестратора фактом:**
- Арифметика сходится: DejaVu «Sales Manager» @8pt = 58.9pt; Inter @12pt = 84.3pt → кегль **8→12**.
- `12` = `typography.scale.small` темы `premium` (реальное значение); `8` = дефолтный фоллбэк в коде.
- **pdf-002 (`e4f2a49`) НЕ причина** (git show --name-only): тронул ТОЛЬКО `image_block.py` +
  `context.py` (там лишь добавлен warning, резолюция токенов байт-в-байт та же) + тест.
  `table_renderer.py`/`stats_grid.py` не касался. Резолюция `typography.scale.small`→12 одинакова
  до и после pdf-002.
- `table_renderer` использует путь `typography.scale.small` с v0.1 (`fccbd50`) — путь никогда не
  был битым. premium имеет `small:12` с создания темы (`9a4e6cc`).
- **Настоящая причина:** `pdf-001` починил рендер premium (до него — пустой PDF/0 страниц из-за
  исключения шрифта). Первый КОРРЕКТНЫЙ рендер premium показал 12pt-ячейки, переполняющие узкие
  4-колоночные таблицы. «Старый PDF на 8pt» — это была другая тема (premium до pdf-001 не
  рендерился вовсе). Баг раскладки **проявлен** батчем, не введён; конкретного коммита-регрессии нет.

**Свип «что ещё переполняется» (пункт владельца):** компоненты со `scale`-токенами в стеснённой
